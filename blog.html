AI PM Framework

Agentic Product Discovery Framework

Most teams jump from idea to prototype and only later realize the workflow should never have been automated. This framework prevents that waste by scoring each candidate workflow before engineering work starts.

Discovery Scoring Model

  • Task Frequency: How often does the user perform this workflow per week?
  • Failure Cost: What is the business impact when the task is done poorly or slowly?
  • Automation Fit: Is the task mostly pattern-based, rules-heavy, or judgment-heavy?
  • Data Readiness: Do you have reliable data sources and clear ground-truth labels?
  • Risk Surface: What is the security, privacy, compliance, and brand exposure level?

Execution Steps

  • Map top 20 user workflows and rank by pain, frequency, and monetization potential.
  • Select top 3 candidates and run two-week shadow pilots with humans still in control.
  • Track completion rate, correction rate, and time saved per attempt.
  • Promote only workflows that beat baseline quality and stay within risk thresholds.

Prioritization Framework

RICE+ for AI Products

Classic RICE misses core AI realities. A feature can have high reach and impact but still fail due to hallucination cost, low reliability, or governance drag. RICE+ adds those factors directly into prioritization.

RICE+ Formula

  • Base: Reach x Impact x Confidence / Effort.
  • Reliability Multiplier: Penalize low pass-rate outputs.
  • Governance Multiplier: Penalize high regulatory review burden.
  • Ops Cost Multiplier: Penalize high inference, monitoring, and support load.

How to Use It

  • Force each proposal to include expected offline eval score and human override rate.
  • Adjust confidence based on observed drift risk and dataset stability.
  • Re-rank quarterly so model and cost improvements can unlock previously rejected bets.

Claude Skills Framework

Skill Portfolio Strategy for PM Teams

Teams often build many prompts but few reliable workflows. A skill portfolio gives PM, design, and engineering a shared operating layer that improves consistency and output quality.

Portfolio Structure

  • Discovery Skills: Interview synthesis, competitor teardown, opportunity tree drafting.
  • Definition Skills: PRD drafting, acceptance criteria drafting, risk logging.
  • Delivery Skills: sprint planning support, release notes, go-to-market briefs.
  • Learning Skills: experiment readouts, KPI diagnosis, postmortem generation.

Governance and Quality

  • Version skills with owner, update date, and known failure modes.
  • Attach examples of good input format and bad input format for each skill.
  • Measure skill adoption, reuse, and downstream document quality over time.

Execution Framework

Prompt-to-PRD Operating Framework

AI can accelerate PRD writing, but only when it is constrained by decision history and product context. This framework turns prompt-driven drafts into reviewable product specs.

Operating Loop

  • Step 1: Frame the problem and define non-goals before drafting requirements.
  • Step 2: Generate candidate PRD sections with explicit assumptions.
  • Step 3: Run a structured critique pass for ambiguity and missing edge cases.
  • Step 4: Record all accepted and rejected decisions in a decision log.
  • Step 5: Publish only after design, engineering, and compliance sign-off.

Success Metrics

  • Cycle time from problem statement to approved PRD.
  • Number of requirement changes after engineering kickoff.
  • Defect leakage attributed to unclear requirements.

Measurement Framework

North Star Metrics for Copilot Products

Raw usage is a weak indicator for AI products. Copilots can be widely tried but rarely trusted. The right north star captures sustained value creation, not novelty clicks.

Metric Stack

  • Primary: Weekly successful assisted tasks per active user.
  • Quality: Acceptance rate, edit distance, and user correction burden.
  • Trust: Repeat use in high-stakes workflows and escalation rate.
  • Business: Time saved, throughput increase, or error reduction by workflow.

Instrumentation Tips

  • Log both assistant suggestion and final user-submitted output.
  • Segment by task difficulty to avoid inflated performance from easy tasks.
  • Track quality decay over time to catch model drift early.

Experimentation Framework

Eval-First A/B Testing for LLM Features

A/B tests alone are too noisy for fast-moving LLM iterations. Start with offline evals to filter bad options, then run online experiments only on strong candidates.

Two-Stage Validation

  • Offline Stage: Evaluate groundedness, instruction following, and policy compliance.
  • Online Stage: Measure task completion, user trust, and measurable business impact.
  • Guardrail Stage: Monitor toxicity, data leakage, and unsafe action suggestions continuously.

Practical Rules

  • Do not launch variants that fail offline safety and quality thresholds.
  • Use fixed benchmark sets so weekly changes remain comparable.
  • Treat regressions in critical flows as launch blockers, not minor issues.

Design Framework

AI UX Reliability Framework

AI UX quality is less about visual polish and more about predictable recovery. Users forgive imperfect outputs if correction and verification are fast and explicit.

Reliability Patterns

  • Show source context and confidence cues where decisions matter.
  • Make outputs editable by default with clear provenance.
  • Provide one-click actions for retry, refine, and escalate to human support.
  • Design empty states and failure states as first-class product experiences.

Design Review Checklist

  • Can users verify critical claims in under 10 seconds?
  • Can users recover from a wrong answer without restarting the workflow?
  • Are limitations explained before the user hits failure?

Design System Framework

Design Tokens for Generative Interfaces

Generative interfaces need token systems that account for dynamic states such as confidence, citations, moderation flags, and collaborative edits.

Token Architecture

  • Response Tokens: typography, spacing, and hierarchy for long-form output.
  • State Tokens: trusted, uncertain, flagged, and needs review states.
  • Interaction Tokens: prompt input, refine action, and citation hover behavior.
  • Safety Tokens: warning and escalation color systems with accessibility contrast.

Implementation Guidance

  • Connect component states directly to eval outcomes where possible.
  • Test token behavior across long answers, short answers, and streamed output.
  • Standardize citation and disclaimer styles across all assistant surfaces.

Governance Framework

Human-in-the-Loop Escalation Framework

Not every workflow should be fully automated. A strong escalation framework defines where human review is mandatory, optional, or bypassed based on risk.

Escalation Tiers

  • Tier 1: Low-risk tasks with auto-approval and periodic audits.
  • Tier 2: Medium-risk tasks requiring spot checks and exception handling.
  • Tier 3: High-risk tasks requiring explicit human approval before action.

Policy Design

  • Define thresholds using monetary impact, legal exposure, and customer harm.
  • Log every escalation event with reason code and final outcome.
  • Review threshold settings monthly to reduce unnecessary friction.

Platform Framework

Model Routing Decision Framework

One model cannot optimize for every workload. Routing lets you match tasks to the right model profile and keep performance within cost and latency constraints.

Routing Dimensions

  • Complexity: reasoning depth and context-window requirements.
  • Latency: acceptable response time by user workflow.
  • Cost: token budget limits by segment or account tier.
  • Risk: safety-sensitive requests routed to stricter pipelines.

Architecture Pattern

  • Start with a lightweight classifier to label intent and complexity.
  • Apply deterministic routing rules with fallback for timeout or safety block.
  • Track per-route quality and cost so routing policy can be tuned weekly.

Team Framework

PM x Design x Eng AI Cadence

AI products fail when discovery, UX, and implementation drift apart. A shared weekly cadence keeps assumptions aligned and speeds learning cycles.

Weekly Rhythm

  • Monday: prioritize top user pain and define success criteria.
  • Tuesday: co-design prompts, system behavior, and failure-handling UX.
  • Wednesday: prototype and run internal dogfood eval set.
  • Thursday: review metrics, failure clusters, and compliance notes.
  • Friday: decide ship, hold, or iterate with explicit rationale.

Cadence Outputs

  • Single shared decision log across PM, design, and engineering.
  • Updated prompt/system version history with owner accountability.
  • Top risk list and mitigation owner for next sprint planning.

Growth Framework

Activation Framework for AI Features

Most AI launches fail at activation, not awareness. Users try the feature once, get mixed output quality, and never return. A structured activation program increases sustained use.

Activation Playbook

  • Use guided first-run experiences with user-specific examples.
  • Offer templated prompts tuned for high-success early workflows.
  • Trigger contextual coaching when confidence is low or edits are high.
  • Unlock advanced capabilities only after base usage competency is established.

KPI Stack

  • Day-7 repeat use in target workflow.
  • First-week success ratio and correction burden.
  • Conversion from assisted trial to durable weekly usage.

Enterprise Framework

Compliance-by-Design for AI Roadmaps

Compliance reviews are often treated as end-stage blockers. Moving policy checks into roadmap planning reduces rework and shortens launch cycles in regulated industries.

Roadmap Integration Model

  • Tag each initiative by data sensitivity and regulatory exposure at intake.
  • Attach required controls and review gates directly to epics.
  • Plan compliance artifacts in sprint scope, not as post-build documentation.
  • Use go-live criteria that combine product KPIs and policy pass criteria.

Operational Benefits

  • Fewer late-stage surprises and less launch churn.
  • Improved executive confidence in AI roadmap commitments.
  • Clear audit trails for decisions, controls, and model behavior changes.

AI Evals Framework

AI Evals That PM Teams Can Actually Run

AI evals should not be an isolated research exercise. PM teams need a lightweight, repeatable operating system that turns quality measurement into product decisions. This article outlines a practical setup that works for small teams and scales to enterprise governance.

Step 1: Define Product-Critical Behaviors

  • List the top 5 user jobs where AI quality directly impacts trust and business value.
  • Translate each job into measurable behaviors such as grounded response, action correctness, or format adherence.
  • Assign a severity score for failure: minor annoyance, workflow block, legal risk, or financial impact.

Step 2: Build a Tiered Eval Suite

  • Smoke Set: 20 to 30 fast checks for core regressions before every release.
  • Quality Set: 100+ representative tasks scored weekly for quality trends.
  • Safety Set: adversarial prompts for policy violations, leakage, and harmful outputs.
  • Golden Set: high-value production-like examples owned jointly by PM, design, and domain experts.

Step 3: Score with Human and Automated Signals

  • Combine rubric-based human scoring with deterministic checks for structure and policy.
  • Track precision and recall of evaluator prompts to avoid misleading pass rates.
  • Log disagreement cases and review them weekly to improve rubric clarity.

Step 4: Connect Evals to Launch Gates

  • Define minimum pass thresholds by workflow risk level.
  • Block release when critical-category quality regresses beyond tolerance.
  • Require mitigation plan and re-test before rollout resumes.

Step 5: Turn Eval Results into Roadmap Inputs

  • Cluster failures by root cause: retrieval quality, prompt design, model limitation, or UX confusion.
  • Create explicit backlog items with owner, severity, and expected quality lift.
  • Re-run targeted eval slices after fixes to verify improvement before closing work.

Common PM Pitfalls and Fixes

  • Pitfall: chasing one aggregate score. Fix: report by workflow and severity tier.
  • Pitfall: running evals only pre-launch. Fix: schedule weekly and pre-release runs.
  • Pitfall: no ownership model. Fix: assign PM owner for eval health and action tracking.
