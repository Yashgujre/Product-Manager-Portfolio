Side Build

Healthcare Policy Copilot

How I built a production-style AI workflow that converts policy text into execution-ready plans with explicit reliability and risk controls.

Yash Gujre

Product Manager · 14 min read · Updated March 2026

Healthcare policy documents are often long, ambiguous, and hard to execute under time pressure. This side build was designed around a single outcome: convert policy text into structured, execution-ready action plans with explicit uncertainty handling.

Live App

Run the product end to end.

Open Healthcare Policy Copilot

Project Identity

  • Transforms policy text into an ordered action plan.
  • Surfaces compliance risk through flags, evidence, and missing information.
  • Uses confidence scoring with deterministic caps.
  • Implements strict contracts, retry logic, and guardrails to keep output reliable.

Public entrypoints:

  • Frontend: `https://yashgujre.github.io/AI-Demos/`
  • Backend API: `https://ai-demos-policy-copilot.vercel.app/api/generate`

Why This Exists

Policy review teams need more than summarization. They need operational clarity:

  • clear next steps
  • owner mapping
  • due windows
  • conflict detection
  • auditability

API Contracts

Endpoint:

  • `POST /api/generate`

Request fields:

  • `policy_text`
  • `domain` (fixed to `healthcare` in v1)
  • `urgency`
  • `organization_context`
  • `requester_role`

Response fields:

  • `summary`, `priority_level`, `actions[]`
  • `compliance_flags[]`, `missing_information[]`
  • `stakeholder_update_draft`, `confidence_score`, `audit_log`

Error contract:

  • `RATE_LIMITED`
  • `VALIDATION_FAILED`
  • `MODEL_ERROR`
  • `BAD_REQUEST`

System Flow

  1. Frontend submits `POST /api/generate`.
  2. Backend enforces method, content type, payload size, CORS, and rate limits.
  3. Zod validates input schema; sanitizer removes likely injection patterns.
  4. Prompt composer builds model input.
  5. Gemini (live) or deterministic mock generates output.
  6. JSON parse + output schema validation run.
  7. If invalid, one correction retry runs.
  8. Guardrails apply conflict detection, confidence caps, and caution prefixes.
  9. Structured response is returned to UI.

AI-Native Implementation Choices

  • Schema-first workflow: model output is accepted only if it satisfies typed contracts.
  • One corrective retry path if parse/schema validation fails.
  • Post-model guardrails are deterministic and separate from the model adapter.
  • Live + mock modes are both first-class to separate capability checks from provider variance.
  • Confidence is policy-driven (caps + caution prefix), not just model-reported.
  • Error behavior is contract-based so frontend handling stays stable across model changes.

Core Capabilities

  • Policy-to-action transformation with ordered steps, owner role, and due window.
  • Compliance signaling with evidence and severity levels.
  • Missing-information surfacing for incomplete policy inputs.
  • Stakeholder update draft to accelerate cross-functional communication.
  • Audit log for constraints detected and assumptions made.
  • Public demo controls: session caps, IP caps, BYOK path, and explicit failure semantics.

Architecture

The architecture is intentionally layered so each reliability concern is independently testable:

  • Frontend (`frontend/src/App.tsx`) for interaction model, validation UX, and result rendering.
  • API handler (`backend/api/generate.mjs`) for gatekeeping, CORS, rate-limits, BYOK extraction.
  • Service layer (`backend/lib/service.mjs`) for orchestration, sanitize, prompt, retry, guardrails.
  • Model adapter (`backend/lib/model.mjs`) for live/mock execution and timeout behavior.
  • Guardrails (`backend/lib/guardrails.mjs`) for conflicts, confidence capping, and caution signals.
  • Shared contracts (`shared/schema.mjs`) used across validation, API, and tests.

Evaluation Harness (Why This Works on Non-Deterministic Models)

The model is non-deterministic, but release confidence is made deterministic through contract-based checks and scenario expectations.

Run deterministic checks:

npm run evals

Checks include:

  • schema validity
  • required field presence
  • missing-information behavior
  • conflict-flag behavior
  • confidence-threshold behavior

Additional checks:

  • `npm run test:sanitize` for input sanitization behavior.
  • `npm run test:guardrails` for conflict extraction and confidence caps.
  • `npm run evals -- --live` for provider-integrated timing and output diagnostics.

Security Notes

  • No provider keys are embedded in frontend bundles.
  • BYOK keys are request-scoped and not persisted in server logs/state.
  • Payload-size and content-type checks reduce abuse and malformed traffic.
  • CORS allowlist blocks unauthorized browser origins by default policy.
  • Rate-limit controls (IP/day + session) govern public usage costs and abuse.

Deployment

  • Frontend deployed to GitHub Pages via workflow.
  • Backend deployed as Vercel serverless endpoint (`/api/generate`).
  • Env controls model mode, timeout, API base URL, CORS origins, and rate limits.
  • Frontend and backend are independently deployable for faster iteration.

Product Operating Decisions

1) Discovery

Define the operating problem as execution quality, not answer fluency.

  • mapped policy-review pain into actionability gaps and risk visibility gaps
  • defined success around time-to-action and reviewability
  • set failure criteria: contradictions, missing fields, and false certainty

2) Definition

Lock interfaces and scope before optimization.

  • froze input/output/error contracts
  • fixed v1 domain to healthcare for higher precision
  • defined deterministic post-processing requirements

3) Design

Design outputs for reviewer decisions, not passive reading.

  • action cards include ownership and due windows
  • compliance flags include evidence and severity
  • audit log and confidence cues support human oversight

4) Build

Convert reliability assumptions into explicit gates.

  • gatekeeping: method/content-type/size/CORS/rate-limit
  • schema parse + single correction retry path
  • guardrails for conflicts and confidence constraints

5) Validation

Operationalize AI evals as a release prerequisite.

  • deterministic eval harness against scenario fixtures
  • unit-level tests for sanitize and guardrail logic
  • live eval mode to detect provider/runtime drift

6) Launch and Operate

Control trust, risk, and cost in production operation.

  • clear user-facing failure semantics and recovery paths
  • BYOK path for advanced usage without widening public cost exposure
  • independent deployment topology for faster safe iteration

Known Constraints

  • v1 domain fixed to healthcare
  • no authentication in v1
  • in-memory limiter fallback is non-persistent
  • conflict detection is rule-based, not full semantic policy reasoning

Failure Diagnostics

Most frontend `Failed to fetch` cases map to:

  • CORS allowlist mismatch
  • incorrect backend URL
  • backend reachability issues

Other expected failures:

  • content-type mismatch (`415`)
  • daily rate cap (`429`)
  • invalid BYOK key (`BAD_REQUEST`)
  • provider timeout (`MODEL_ERROR`)

Closing

Healthcare Policy Copilot was built as an AI workflow system, not a prompt demo. The impact comes from PM decisions across the full PDLC: bounded scope, explicit contracts, deterministic guardrails, eval-first validation, and production-ready operations design.
