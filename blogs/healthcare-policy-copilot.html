Side Build

Healthcare Policy Copilot

A production-style AI system that transforms healthcare policy text into structured, execution-ready action plans.

Yash Gujre

Product Manager · 14 min read · Updated March 2026

Healthcare policy documents are often long, ambiguous, and difficult to operationalize under time pressure. This build focuses on one job: convert messy policy and SOP text into a deterministic action plan with ownership, urgency, compliance visibility, and confidence signaling.

Open the deployed product and run scenarios end to end.

Open Healthcare Policy Copilot

Project Identity

  • Transforms policy text into a structured execution plan with ordered actions.
  • Surfaces compliance risk via flags, evidence, missing information, and confidence score.
  • Enforces output reliability using schema validation, retry logic, and deterministic guardrails.
  • Frontend on GitHub Pages and API backend on Vercel.

Why This Exists

Policy reviewers and operations teams need more than summaries. They need clear next steps, owner mapping, due windows, conflict detection, and auditable assumptions. The product closes the gap between policy reading and policy execution.

Product Behavior

Input includes policy text, urgency, organization context, and requester role (healthcare domain in v1). Output includes summary, priority, ordered actions, compliance flags, missing information, stakeholder draft, confidence score, and audit log.

System Flow

  1. Frontend sends `POST /api/generate`.
  2. Backend enforces method, content-type, payload size, CORS, and rate limits.
  3. Input is validated with Zod, then sanitized for prompt-injection patterns.
  4. Prompt is composed and sent to Gemini (or deterministic mock mode).
  5. Response JSON is parsed and validated; one correction retry is attempted if invalid.
  6. Guardrails enforce conflict detection and confidence caps.
  7. Final structured response is returned to the UI.

Repository Structure

The architecture cleanly separates frontend, backend orchestration, model adapter, guardrails, shared contracts, tests, and eval harnesses. Contracts live in `shared/schema.mjs`, orchestration in `backend/lib/service.mjs`, and end-user rendering in `frontend/src/App.tsx`.

API Contract

The endpoint is `POST /api/generate`. The request accepts `policy_text`, `domain`, `urgency`, `organization_context`, and `requester_role`. The response returns typed fields including `actions[]`, `compliance_flags[]`, `missing_information[]`, and `audit_log`.

Errors follow a stable contract: `RATE_LIMITED`, `VALIDATION_FAILED`, `MODEL_ERROR`, or `BAD_REQUEST` with consistent messaging.

Backend Reliability Design

  • Strict request gatekeeping and content checks.
  • Configurable CORS allowlist with explicit browser-block behavior for mismatches.
  • BYOK support via request header without persisting keys.
  • Timeout-aware model adapter using `AbortController`.
  • Single corrective retry when output parse/schema validation fails.
  • Deterministic guardrails for conflict detection and confidence capping.

Guardrails and Risk Controls

The guardrail layer extracts temporal constraints such as immediate actions, wait conditions, and deadline windows. If conflicting instructions are detected, severity is escalated and confidence is capped. When critical details are missing, confidence is reduced and caution is explicitly surfaced in summaries.

Frontend UX

  • Scenario pills for quick demonstrations.
  • Inline validation for key fields.
  • Structured output panels for actions, flags, missing info, and stakeholder draft.
  • Audit log visibility and error diagnostics.
  • Session-cap behavior for public demos, with BYOK bypass for power users.
  • Accessibility-first form labeling, live regions, and alert semantics.

Quality and Evaluation

The project includes deterministic eval scenarios (`npm run evals`) and focused unit tests for sanitization and guardrails. Live eval mode adds timing checks and richer failure diagnostics.

Local Runbook and Deployment

Local development runs frontend and Vercel API separately. Deployment splits frontend (GitHub Pages) and backend (Vercel). Environment variables control mode, API base URL, rate limit caps, model timeouts, and CORS origin policy.

Known Constraints

  • v1 domain is fixed to healthcare.
  • No authentication in v1.
  • In-memory rate-limit fallback is non-persistent.
  • Conflict detection is rule-based, not full semantic reasoning.

Failure Mode Diagnostics

Most frontend fetch failures resolve to CORS allowlist mismatch, incorrect backend URL, or backend reachability. Additional known failure states include content-type mismatch (415), daily rate cap (429), invalid BYOK key (`BAD_REQUEST`), and provider timeout.

Positioning

Healthcare Policy Copilot is designed as a real AI workflow system, not a generic chatbot. It demonstrates production discipline through strict contracts, guardrails, controlled failure behavior, deterministic eval coverage, and deployment-ready architecture.
