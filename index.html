How I think about product

01

Start with the economic model

Before writing specs, map how value is created and captured. Features without a clear ROI path get cut — or never get adopted.

At Capital One, we modeled conversion lift and default reduction before writing a single API spec. That constraint shaped every priority in the backlog.

02

Compliance is architecture, not a gate

HIPAA, SOC 2, financial regulation — these belong in the data model on day one, not in a remediation sprint before launch.

The care coordination platform was built with HIPAA audit retention as a data model primitive from day one — not a reporting layer bolted on before go-live.

03

Build for interoperability by default

Every platform decision should make the next integration cheaper. API-first design is how you avoid rebuilding the same thing five times.

04

Instrument before you ship

If you can't measure a handoff, a decision, or a workflow step — you can't improve it. Observability is a product requirement, not a post-launch task.

Referral acknowledgment time dropped from 4.2 to 1.1 days because we instrumented each handoff event individually before the first release shipped.

05

Prioritize adoption over capability

A feature no one uses is not a shipped feature. Workflow fit, training, and change management are part of the product surface — not someone else's problem.

06

Disaggregate the metric

Aggregate KPIs hide the signal. Break cycle time, conversion, or cost into its components — the improvement always lives in one specific step, not the average.

07

Make the constraint explicit

Every backlog has an implicit constraint driving prioritization. Surface it. A team that knows the binding constraint makes better tradeoffs without asking for permission on each one.

At Galderma, the binding constraint was Oracle Fusion data latency — not feature scope. Making that explicit cut two sprint cycles of misdirected backlog work.

08

Explainability is a feature, not a footnote

In regulated environments, a decision no one can audit is a liability. Explainability drives adoption, reduces risk review cycles, and earns operator trust over time.
